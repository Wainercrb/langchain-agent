Get Enrollment API

Introduction ¶

The Marketplace Consumer Record (MCR) Get Enrollment Application Program Interface (API) will be leveraged by Enhanced Direct Enrollment (EDE) Entities to give EDE Issuers and EDE Web Brokers the ability to retrieve and display Enrollment data to the Consumer or EDE Representative through the EDE Entity. The goal of EDE is to provide a Consumer with a complete Eligibility and Enrollment service on the EDE Partner site, without ever having to navigate to the Federally Facilitated Marketplace (FFM). This service will allow users to view Insurance Plan Policies (IPPs) and Pended Plan Selections (PPSs) on the EDE Entity site, aligning with the overall goals of EDE. EDE Entities will be able to retrieve a Consumer’s Enrollment data by sending a request with an Insurance Application Identifier. The API will return all Enrollment data tied to that Application that the requester has been authorized and received permission to view. EDE Entities will be able to request Enrollment data for consumers once an IPP or PPS has been created. Depending on the type of EDE Entity using the API, a limited amount of Enrollment data may be returned.

Current System ¶

EDE Entities will leverage the MCR Get Enrollment API, with some additional enhancements. The MCR Get Enrollment API was designed to provide real-time access to FFM policy data. This API also utilizes the Data Services Hub (HUB) and the Health Insurance Oversight System (HIOS) to validate that policy data pertaining to a specific Issuer is only returned for requests from that Issuer.

The API will also replace the functionalities of the Fetch Eligibility service utilized by current DE Partners. This service returns a limited amount of eligibility and Enrollment data to the DE Partners.

Functional Description - Current System ¶

The Fetch Eligibility Services allows DE Entities to retrieve a subset of Enrollment and application data from the FFM. This service is used to populate an Enrollment request prior to submit Enrollment.

User Community Description ¶

The current users of the MCR Get Enrollment API are the community of Issuers that provide health and dental coverage for the FFM. This includes both DE and non-DE Issuers.

The current users of the Fetch Eligibility Service are the community of DE Partners. This includes both DE Issuers and Web Brokers.

The users of the Get Enrollment API will include A/B, Consumers, and EDE Entities. The table below defines the different types of EDE Entities.

Term Definition
EDE Issuer An Issuer that provides an enrollment experience for Consumers outside of HealthCare.gov.
EDE Web Broker An Entity that sells plans and policies from multiple issuers across the Marketplace and provides an enrollment experience for Consumers outside of HealthCare.gov.

Overview and Scope ¶

Functional Description ¶

The EDE Enrollment functionality will allow EDE Representatives to retrieve Consumer Enrollment data using an Insurance Application Identifier as the request option. The EDE Enrollment API will grant EDE Representatives and EDE Consumers the ability to see all Enrollment data for that Consumer, provided several restrictions. If retrieving policies through a Web Broker, all policies and pended plans tied to the requested Application will be returned. If retrieving policies through an Issuer, only policies and pended plans covered by that Issuer tied to the requested Application will be returned.

The API will also include the capability to return a limited view of a Consumer’s Enrollment data. If a Consumer with an existing policy is trying to create a new policy with an Issuer they were not previously enrolled with, but have not created a new Application, a “limited” view of the Enrollment data tied to that Application will be returned. This will allow the enrolling Issuer to create a similar policy to the one the Consumer was enrolled in previously, without revealing any proprietary or sensitive information about the previous policy.

Assumptions ¶

The following assumptions were used in the design of this API:

  1. The MCR Get Enrollment API will be used to return Enrollment data to EDE Entities, with some additional enhancements. A new API will not be built.

    a. Implementation of these enhancements may take place across multiple sprints and releases.

    b. User story assignment to sprints and releases defines when certain parts of this functionality will be implemented.
  2. This document provides the Enrollment data functionality and technical design as it relates to API, not User Interface (UI).
  3. EDE Entities will leverage the Insurance Application Identifier in order to request Enrollment Data.
  4. The IPP contains the same information as Pre-Audit, and therefore the Get Enrollment API will not pull any information from the Pre-Audit file at this time.
  5. In order for Enrollment data to be available to an EDE Entity, there must be a submitted application for the consumer that also generates an IPP or PPS.
  6. EDE Entities should not be able to see superseded policy segments.
  7. Issuers will only be able to view full Enrollment data for policies that contain HIOS IDs that fall under a single FEIN. If Issuers have multiple FEINs, one Partner-ID must be created for each FEIN.
  8. Web brokers will be able to view all policies tied to a Consumer if they have been granted Permission to view and edit the Application that created the policy, regardless of the enrolling Issuer.
  9. The Hub will handle authentication for EDE Entities making requests to MCR.
  10. The functional design is subject to change based on technical design.

Constraints ¶

Enrollment Data is pulled from IPP and PPS documents that reside within the Federal Facilitated Marketplace (FFM). As such, the following constraints apply:

  1. EDE Entities will not be able to retrieve any data that resides solely on Outbound 834s or the Pre-Audit file.
  2. Changes to the IPP or PPS data structure in the FFM database may impact this API. There are no anticipated changes at this time.
  3. All policies tied to a requested Application Identifier will be returned with each request. This may include multiple policies that include multiple consumers. However, only one Application ID may be requested at one time. Batch Functionalities will not be supported for this API.
  4. This API can only be used to retrieve Enrollment Data. It should not be used to submit an Enrollment.
  5. Timely updates and coordination with cross-marketplace entities is essential to the project’s overall success. Development and implementation of this service is dependent on successful integration between DE entities and MCR.
  6. FFM will not be aware of requests that fail at the Hub.
  7. One Partner ID may be used per request.
  8. The MCR Get Enrollment API can only return a maximum of 100 coverage spans in a given API response.

Dependencies ¶

Succesful use of this API is dependent on succesful connectivity between EDE Entities, the HUB, and the ID Proofing API. Additionally, EDE Entities must meet all neccesary security requirements prior to use of this API.

Design ¶

An EDE Entity will submit a request to the Get Enrollment API to receive policy data for the specified Insurance Application. The EDE Entity will be able to retrieve all policies tied to the application that they have received Permission to view.

There are two types of entities that may make a request to the Get Enrollment API – an Issuer and a Web Broker. Entities must indicate if they are an Issuer or a Web Broker when On-Boarding to the HUB. The HUB will then share this information with the FFM for each request sent to the API. Once the request is sent, the API will check if the requesting entity has “Permission” to view the policies tied to the Application that they are requesting by sending the Application ID and the Partner ID to the Permission API. If an Issuer makes a request with an identifier other than Application ID, then Permission will not be verified. Only IPPs that contain a HIOS ID of the requesting Issuer will be returned, as is consistent with the current functionality of the Issuer Flow of the Get Enrollment API. If a Web Broker makes a request with an identifier other than Application ID, an error will be thrown and no Enrollment data will be returned.

If the requesting EDE Entity is an Issuer, and the Issuer is granted permission to view the requested policy, the API will retrieve the HIOS IDs tied to the FEIN of the Issuer using a pre-existing HIOS service. This will retrieve all relevant policies tied to the requested Insurance Application from the FFM database, filtering out any policies with HIOS IDs that do not fall under that FEIN.

If the requesting EDE Entity is a Web Broker, and the Web Broker is granted permission to view the requested policies, all policies tied to the Application will be returned. The API will not make an additional call to HIOS.

As is in alignment with the prior design of the Get Enrollment API, prior versions of policy segments and superseded segments will not be returned.

Until the end of Open Enrollment, no policies created through Batch Auto Re-enrollment (BAR) for the following coverage year will be returned in the API response.

EDE Limited Enrollment Service ¶



When a request is made with Application ID, and if the requesting Entity is an Issuer, internal logic will be performed to verify if policies being returned are enrolled with that Issuer. If any of the policies were enrolled with an Issuer other than the requesting Issuer, Limited Enrollment information from those policies will be returned. This allows the EDE Representative to align a Consumer’s new policy as closely as possible to their prior policies.

If a Consumer was previously enrolled with the requesting Issuer and a different Issuer within the same coverage year, the full policy for that Issuer will be returned, and a limited set of policy data for the other Issuer will be returned.


This will be accomplished via internal logic, and will not be a separate service or request.
